# NijiArchive プロジェクト総合ロードマップ Part 1
## コンセプト・要件定義・ペルソナ設計

---

## 0. このドキュメントの位置づけ

結論：このプロジェクトは「学習目的の個人開発」だが、マネタイズしないからこそ、コンセプト・要件定義・設計を本気でやる価値がある。

理由：
- 実務では「予算」「納期」「政治」が設計を歪めることが多い
- 個人開発は「正しい順番」で進める練習ができる稀有な機会
- コンセプト→要件定義→設計→実装→運用の全工程を1人で通しで経験することが、シニア〜スタッフエンジニアに求められる「一気通貫で設計判断できる力」の土台になる

だからこのロードマップは、要件定義や設計フェーズを「実装の前座」として軽く扱わず、Step 0〜8と同格の重みを持たせて構成する。

---

## 1. プロジェクトコンセプト

### 1-1. プロジェクト名

**NijiArchive（仮）**

### 1-2. 原点（Why）

- 虹ヶ咲のライブ配信で数万人同時視聴を体験した
- アーカイブ視聴時に同時ログイン制限が発動し、「これどうやって実現してるんだ」という疑問が生まれた
- 動画配信の基礎（Section 0〜7）を体系的に学んだ結果、「作りながらさらに深く理解したい」という欲求に発展した

### 1-3. コンセプトステートメント

> 表向きは「ニジガク系ライブのアーカイブ・関連動画をまとめるファンポータル」。
> その裏側では、配信技術の中身（ビットレート推移、ABR切り替え、CDNキャッシュ状態、視聴統計のリアルタイム集計）が丸ごと覗ける。
> ファンにはファンサイトとして、エンジニアには「配信システムの生きた教材」として機能する、二層構造のサービス。

### 1-4. 二軸の役割分担

| 軸 | 対象 | 目的 | 主な機能 |
|----|------|------|---------|
| 軸1（表）：ファンポータル | ニジガクファン、一般視聴者 | 集客、エンタメ価値の提供 | YouTube API連携での関連動画集約、アーカイブ視聴、コメント、盛り上がり検出ハイライト |
| 軸2（裏）：配信可視化ダッシュボード | 訪問したエンジニア、採用担当者、技術コミュニティ | 技術力のアピール、学習の実演 | ビットレート/ABRのリアルタイム表示、CDNキャッシュ状態、視聴統計の集約 |

両軸は独立した機能ではなく、**同じ配信基盤の「見せ方」が違うだけ**という設計にする。裏側のダッシュボードはトグルで表示/非表示を切り替えられ、通常の視聴体験を邪魔しない。

### 1-5. なぜこの二軸が成立するか（設計思想）

- 単なる「Netflixクローン」「Twitchクローン」だと、車輪の再発明にしかならず、差別化ポイントがない
- 「学習目的で作った」という事実を隠さず、**むしろそれをコンテンツにする**ことで、模倣ではなく作品になる
- エンジニアが自分のポートフォリオとして人に見せたとき、「配信サイトを作りました」より「配信の仕組みが透けて見えるサイトを作りました」の方が、会話のきっかけになり、技術力の証明として機能する

### 1-6. コンセプトの非目標（Out of Scope）

明確にしないと肥大化するので、最初にやらないことを決めておく。

- マネタイズ（広告収益化、課金機能）は将来検討事項とし、MVPには含めない
- YouTube本編の動画を無断で再配信することはしない（著作権侵害になるため）。あくまで公式リンクの集約、メタデータの分析、コメント盛り上がり分析に留める
- 複数アイドルグループ・複数コンテンツへの横展開は最初は考えない（ニジガクに閉じる）
- ネイティブアプリ化は考えない（Webブラウザ完結）
- 決済機能は実装しない

## 2. 要件定義

### 2-1. ステークホルダー（このプロジェクトにおける関係者）

| ステークホルダー | 役割 | 関心事 |
|-----------------|------|--------|
| 開発者（yhok） | 企画・設計・実装・運用のすべて | 技術力の証明、学習の深化、転職活動でのアピール材料 |
| ファン訪問者（想定） | エンドユーザー | 見やすいアーカイブ、話題のハイライトがすぐ分かる |
| エンジニア訪問者（想定） | 技術的な評価者 | 裏側の技術がどう作られているか、設計判断の妥当性 |
| YouTube（外部プラットフォーム） | データ提供元 | API利用規約の遵守、著作権保護 |

### 2-2. ペルソナ

**ペルソナA：ファン訪問者「みさき（24歳、会社員）」**
- 虹ヶ咲のライブ配信を追っているが、全部の配信をリアルタイムで見きれない
- 「あのライブのあの瞬間、もう一回見たい」というニーズがある
- SNSで話題になった切り抜きから元配信を辿りたい
- 技術的なことには興味がなく、UIがシンプルで見やすいことが重要

**ペルソナB：エンジニア訪問者「たかし（29歳、Webエンジニア）」**
- yhokのポートフォリオサイトや技術記事経由でNijiArchiveに辿り着く
- 配信技術に詳しくないが、興味はある
- 可視化ダッシュボードを見て「へえ、こうやって動いてるんだ」という驚きを得たい
- GitHubリポジトリのコードやREADMEも見る可能性がある

**ペルソナC：採用担当者・面接官「田中さん（38歳、エンジニアリングマネージャー）」**
- 転職活動の一環でポートフォリオとして紹介される
- 「なぜこの設計にしたか」を語れるかを見ている
- 実際に動くサービスとして触ってみて、UXの完成度も評価対象にする

### 2-3. 機能要件（Functional Requirements）

#### FR-1：動画アーカイブ視聴（軸1）
- FR-1-1：管理者がMP4をアップロードし、HLS形式に自動変換する
- FR-1-2：視聴者はブラウザ上でHLS動画を再生できる
- FR-1-3：複数解像度（1080p/720p/360p等）のABR切り替えに対応する
- FR-1-4：視聴履歴が記録される

#### FR-2：認証・視聴権限（軸1）
- FR-2-1：ユーザー登録・ログイン機能を提供する
- FR-2-2：同一アカウントの同時ログインを1セッションに制限する
- FR-2-3：ログインしていないユーザーは視聴できない、または制限された範囲のみ視聴できる

#### FR-3：コメント機能（軸1）
- FR-3-1：視聴中の動画にリアルタイムでコメントを投稿できる
- FR-3-2：コメントは動画のタイムスタンプと紐づく
- FR-3-3：コメント投稿数の急増を検知し、「盛り上がりポイント」として自動マークする

#### FR-4：関連動画・外部連携（軸1）
- FR-4-1：YouTube Data APIを使い、関連する公式動画・切り抜き動画の情報を取得する
- FR-4-2：取得した動画情報（サムネイル、タイトル、公開日、リンク）を一覧表示する
- FR-4-3：著作権保護のため、動画本体は埋め込みまたはリンクのみとし、再配信は行わない

#### FR-5：視聴者数表示（軸1）
- FR-5-1：現在の視聴者数を表示する
- FR-5-2：更新間隔は数秒単位とし、リアルタイム性より安定性を優先する

#### FR-6：配信可視化ダッシュボード（軸2）
- FR-6-1：現在再生中のビットレートを表示する
- FR-6-2：ABR切り替えが発生した際にログとして表示する
- FR-6-3：CDNキャッシュのHIT/MISS状態を表示する（可能な範囲で）
- FR-6-4：全視聴者の視聴ビットレート分布を集計し、統計として表示する
- FR-6-5：ダッシュボードはトグルで表示/非表示を切り替えられる

#### FR-7：管理機能
- FR-7-1：管理者が動画をアップロード・編集・削除できる
- FR-7-2：管理者がユーザーの視聴権限を管理できる

### 2-4. 非機能要件（Non-Functional Requirements）

#### NFR-1：パフォーマンス
- NFR-1-1：動画再生開始までのレイテンシは3秒以内を目標とする
- NFR-1-2：コメント投稿から他視聴者への反映まで1秒以内を目標とする
- NFR-1-3：秒間100件のコメント投稿があってもシステムが落ちないこと

#### NFR-2：可用性
- NFR-2-1：個人開発のため24/365稼働は目標としないが、意図的なダウンタイムなく動作すること
- NFR-2-2：エンコード処理の失敗が視聴中の他ユーザーに影響しないこと

#### NFR-3：セキュリティ
- NFR-3-1：認証情報は適切にハッシュ化して保存する
- NFR-3-2：Presigned URLの有効期限を適切に設計し、不正な共有を防ぐ
- NFR-3-3：YouTube APIの利用規約を遵守する

#### NFR-4：保守性・可観測性
- NFR-4-1：主要な処理にログを仕込み、障害発生時に原因を追跡できること
- NFR-4-2：OpenTelemetryによるトレーシングを導入する
- NFR-4-3：コードは第三者（採用担当者含む）が読んでも理解できる構造にする

#### NFR-5：コスト
- NFR-5-1：個人開発の範囲で継続運用可能なインフラコスト（月数千円程度）に収める
- NFR-5-2：Cloudflare R2のエグレス無料特性を活用し、配信コストを抑える

### 2-5. 制約条件（Constraints）

- 開発者は1人（yhok）、業務外の時間で開発する
- 技術スタックはRails 8 + Cloudflareエコシステムを軸とする（学習目的との整合性）
- YouTube API利用規約・著作権法を遵守する
- マネタイズしない前提のため、インフラコストは自己負担できる範囲に抑える

### 2-6. 成功基準（Success Criteria）

このプロジェクトの「成功」は事業的な指標ではなく、以下で判断する。

- 技術記事として5本以上のアウトプットが生まれること
- 面接で「このシステムの設計判断」を一貫して説明できること
- GitHubリポジトリとして第三者が見て理解できる状態になっていること
- 実際にエンジニアコミュニティ（Twitter、勉強会等）で反応を得られること
- 開発者自身が「配信システムの仕組みを腹落ちして理解した」と感じられること

## 3. ユースケース（主要シナリオ）

### UC-1：ファンがアーカイブを見つけて視聴する

1. ファンがSNS経由、または検索経由でNijiArchiveにアクセスする
2. トップページで話題のライブ・ハイライトが一覧表示されている
3. 気になるアーカイブをクリックし、ログイン（or 新規登録）する
4. 動画が再生される。ABRにより回線状況に応じた画質で視聴できる
5. コメント欄で当時の盛り上がりを追体験できる
6. 「盛り上がりポイント」のマーカーをクリックすると、その時間にジャンプする

### UC-2：エンジニアが技術的な観点でサイトを見る

1. エンジニアがyhokの技術記事、GitHub、SNS経由でNijiArchiveにアクセスする
2. 視聴ページ右下の「配信を覗く」ボタン（ダッシュボードトグル）をクリックする
3. リアルタイムのビットレート推移グラフ、ABR切り替えログが表示される
4. 「全視聴者の視聴ビットレート分布」を見て、実際に他の訪問者がどんな回線状況で見ているかを知る
5. 気になった場合、GitHubリポジトリやZenn記事へのリンクから設計判断の詳細を読む

### UC-3：同時ログイン制限が発動する

1. ユーザーAがPCでログインし、動画を視聴中
2. 同じアカウントでユーザーAがスマホからもログインを試みる
3. スマホ側の画面で「他のデバイスで視聴中です」と表示され、視聴がブロックされる
4. PC側の視聴は継続される

### UC-4：管理者が新しいアーカイブを公開する

1. 管理者がライブ配信終了後、録画したMP4を管理画面からアップロードする
2. Solid QueueがバックグラウンドでFFmpeg変換ジョブを実行する（複数解像度）
3. 変換完了後、R2にHLSファイルがアップロードされる
4. 管理者が公開設定をONにする
5. トップページに新着アーカイブとして表示される

## 4. 技術スタック選定とその理由

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| バックエンド | Rails 8 | 業務での主力スタック、Solid Queue/Cacheの学習継続 |
| リアルタイム通信 | Action Cable | 未経験のエコシステムを実践投入、コメント・同時ログイン制限・可視化ダッシュボードの3用途に使える |
| デプロイ | Kamal 2 | ChronosHubで習得済み、追加学習コストが低い |
| ストレージ・CDN | Cloudflare R2 + CDN | エグレス無料でコストメリット大、ChronosHubでの実験知見を活用 |
| DB | PostgreSQL（Solid Queue/Cache込み） | Rails 8標準構成、Redis不要のシンプルな運用 |
| 動画変換 | FFmpeg | Section 0〜7で習得済み |
| ベクトル検索 | pgvector | 将来のAI機能（Step 7）で使用 |
| AI API中継 | Cloudflare AI Gateway | 将来のAI機能で使用 |
| 監視 | OpenTelemetry + Grafana Cloud | Phase 0で未着手だった領域の回収 |
| 外部API | YouTube Data API v3 | 関連動画・メタデータ取得 |

## 5. リスクと対応方針

| リスク | 影響 | 対応方針 |
|--------|------|---------|
| YouTube APIの利用規約違反 | サービス停止リスク | 動画本体の再配信はせず、メタデータ・埋め込みリンクのみ扱う |
| 著作権侵害（原作コンテンツの無断使用） | 法的リスク | アーカイブ動画は自分で用意した素材（自作、または権利者が許諾したもの）を使う。実在のニジガク公式映像を無断使用しない設計にする |
| スコープ肥大化 | 完成しない | Step単位で「完了基準」を明確にし、次のStepに進む前に区切る |
| 個人開発のモチベーション低下 | 中断リスク | 各Stepの完了時に必ず記事化し、達成感を可視化する |
