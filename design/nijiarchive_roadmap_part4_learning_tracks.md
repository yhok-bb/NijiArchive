## 0. このドキュメントの位置づけ

結論：Part 3のStep構成に「深掘りフェーズ」を追加し、Railsエコシステム・ミドルウェア・インフラ・API設計の学習密度を上げるための設計書。Part 1〜3の内容（コンセプト・アーキテクチャ・Step本体）は変更しない。

### 0-1. 採用した方式：進化型

深掘りの組み込み方には3つの選択肢があった。

| 方式 | 内容 | 判断 |
|------|------|------|
| 進化型 | MVPの各Stepはleanなまま作り、Step完了後に深掘りフェーズで読解・置き換え・計測を行う | **採用** |
| 最初から本格構成 | Step自体を実務レベル構成で再設計する | 不採用（動くものが出るまでが遠く、スコープ肥大化と正面衝突） |
| タイムボックス制 | 深掘りは各Step最大1週間で区切る | 不採用（OSS読解を時間で区切ると中途半端になりやすい） |

進化型を選んだ理由：実務で一番価値のある経験は「動いているものを安全に置き換える」ことであり、進化型はその練習を各Stepで強制する。また「完璧より完成」の原則（動くものを先に確保する）とも両立する。詳細は `docs/adr/` の該当ADRを参照。

### 0-2. 運用ルール（3つ）

1. **本体Stepの完了基準を満たすまで深掘りに着手しない**。深掘りが長引いてもプロジェクト本体が人質にならないようにするため
2. **1 Step = 1深掘りテーマ**。そのStepが一番深く刺さる領域を1つだけ割り当てる。全Step全領域を深掘りすると期間が14〜16ヶ月級になり「完成しない」リスクに直撃する
3. **深掘りにも数値・観察可能な完了基準を置く**。「読んだ」「なんとなく分かった」で終わらせない

## 1. 深掘りフェーズの型（5段サイクル）

すべての深掘りは同じ型で回す。実務の「調査→設計→実装→検証→共有」のサイクルそのものを型として練習する。

```
1. 読む      … OSSソース・RFC・公式仕様を読む（code-deep-diveスキルを使う）
2. 記録する  … 読解ノートを残し、設計判断はADR（/adr）に記録する
3. 手を動かす … 置き換え実装 or 計測実験。読むだけで終わらせない
4. 検証する  … 深掘り自体の完了基準（数値・観察可能な事実）を満たす
5. 記事化する … 深掘りこそ記事の主戦場（/step-article）
```

### 1-1. 「読む」にはissueトラッカーも含める（OSS上流へのフィードバック）

深掘り対象のgemを読むときは、ソースと一緒に**使った範囲のissueトラッカーも確認する**。ドキュメントに書かれた挙動と実装の挙動を突き合わせながら使うことで、実利用に基づいた質の高い報告ができる。本プロジェクトの構成では、次の領域が上流への報告が価値を持ちやすい：

- **Solid Queue / Solid Cable / Solid Cache / Kamal / Thruster** — 比較的新しいgemで、実運用からのエッジケース報告が現在も活発（特にconcurrency制御まわり）
- **aws-sdk-s3 × R2** — S3互換ストレージ固有の非互換が顕在化しやすい組み合わせ（Presigned URL・マルチパート・Active Storage direct upload）
- **opentelemetry-rubyのinstrumentation群** — 開発が活発で、計装カバレッジに関する報告の需要が高い

一方、Rails本体の主要経路のような成熟した領域で挙動に違和感があれば、まず自分の使い方を疑う。

**貢献の段階**（難度順）：既存issueへの再現手順の追加 → ドキュメント修正のPR → 不具合報告 → 修正PR。いずれも、最小再現を用意してから行うことが報告の質を左右する。デバッグ時に該当gemのissueを検索することを習慣にする。

## 3. 各深掘りの詳細

### DD-1：CDN・ストレージ実測（Step 1完了後、インフラ系）

Step 1の完了基準「CDNキャッシュのHIT率を計測し数値で示せる」を、一段深い理解に引き上げる。

| 段階 | タスク |
|------|-------|
| 読む | Cloudflare CDNのキャッシュ仕様（Cache-Control、CF-Cache-Statusヘッダ）、HLS仕様（RFC 8216）のマニフェスト/セグメントの性質の違い |
| 記録 | マニフェスト（.m3u8）とセグメント（.ts）でTTLを変える理由をADR化 |
| 手を動かす | TTL別のHIT率実測（curlでCF-Cache-Statusを集計）、キャッシュパージの伝播時間実測、Presigned URLのクエリ文字列がキャッシュキーを割る問題の検証と対策設計。KamalのThrusterが何をしているか（TLS終端・静的ファイル配信・X-Sendfile）を構成図に書き起こす |
| 検証（完了基準） | セグメントのキャッシュHIT率90%以上を実測データで示せる。Presigned URLとCDNキャッシュの両立方式をADRで説明できる |

### DD-2：Authentication Generatorとセッションの内側（Step 2完了後、OSS読解系）

「Deviseを使わず内部実装を理解する」というプロジェクト方針を、生成コードの一段下まで掘る。

| 段階 | タスク |
|------|-------|
| 読む | `rails generate authentication` が生成する全ファイル。その下にある `has_secure_password`（ActiveModel::SecurePassword）、bcryptのコスト設計、`cookies.signed` の実装（ActiveSupport::MessageVerifier）、CSRF対策の仕組み |
| 記録 | 「ログインボタンを押してからCurrent.userが入るまで」のシーケンス図を書く |
| 手を動かす | 署名cookieを改ざんして検証が失敗することを実験で確認。セッション固定攻撃を再現し、防御されていることを確認。生成されたセッション管理を同時ログイン制限（Sessionsテーブル）の要件に合わせて拡張する |
| 検証（完了基準） | 認証フローを資料なしで口頭説明できる（面接想定の3分説明を録音して確認）。改ざん実験・セッション固定実験のログが読解ノートに残っている |

### DD-3：負荷とロックの実測（Step 3完了後、DB・整合性系）

Part 2で確立した「表示用データはAP寄り」の判断を、「知っている」から「計測した」に引き上げる。

| 段階 | タスク |
|------|-------|
| 読む | PostgreSQLのMVCCとロックの種類、Railsのコネクションプールの挙動 |
| 記録 | 同期書き込み設計と非同期永続化設計の比較をADR化（数値付き） |
| 手を動かす | 秒間100コメントの負荷試験を「同期でDB書き込み」版と「Action Cable即配信＋Solid Queue非同期永続化」版の両方に実施し、p95レイテンシを比較。EXPLAIN ANALYZEでインデックス有無の実行計画を比較。ロック競合を意図的に再現（同一行への並行UPDATE） |
| 検証（完了基準） | 同期版と非同期版の性能差がグラフで示せる。comments(video_id, created_at)複合インデックスの効果をEXPLAINの出力で説明できる |

### Step 3.7：API再設計（新設Step、API設計系）

深掘りフェーズではなく本体Step扱い。Step 1〜3で生えたAPI（videos / comments / sessions）が出揃った時点で、「動いているAPIを規約駆動に置き換える」経験をする。進化型アプローチの代表例。

#### 実装タスク

| タスク | 詳細 | 設計判断ポイント |
|-------|------|----------------|
| OpenAPI仕様書化 | 既存APIをOpenAPI 3.x形式で仕様化し、以後spec-firstに逆転する | 仕様と実装の乖離をどう防ぐか（CIでの検証） |
| スキーマ検証のテスト組み込み | committee等でリクエスト/レスポンスをOpenAPI specと突合するテストを追加 | テストの層をどこに置くか |
| バージョニング戦略 | URLパス（/api/v1/）vs ヘッダの比較検討 | 個人開発規模での現実解をADR化 |
| エラーレスポンス設計 | RFC 9457（application/problem+json）準拠に統一 | エラーコード体系の設計 |
| レートリミット | rack-attackのソースを読んだ上で、最小のレートリミッタを自作（Solid Cacheバックエンド） | fixed window vs sliding windowのトレードオフ |
| ページネーション再設計 | offset方式とcursor方式を比較し、コメントAPIをcursor方式に置き換え | 大量データでのoffsetの劣化を実測 |

#### 完了基準

- OpenAPI specと実レスポンスの突合テストがCIで通っている
- 自作レートリミッタが429＋Retry-Afterヘッダを返し、rack-attack相当の動作をするテストがある
- offset/cursorの性能差を10万件規模のデータで実測した数値がある

### DD-4.5：Action Cable / Solid Cableの内部（Step 4.5完了後、OSS読解系）

軸2の心臓部を支える技術の内部を読む。このプロジェクトで一番リアルタイム通信に依存するStepの直後に置く。

| 段階 | タスク |
|------|-------|
| 読む | actioncable gem（Connection・Channel・subscription管理・Worker poolの構造）、Solid Cableのpub/sub実装（DBポーリングでどうブロードキャストを実現しているか）、WebSocketハンドシェイクの流れ |
| 記録 | 「コメント投稿→他ブラウザに表示」の経路をプロセス・テーブルレベルで図解する |
| 手を動かす | Solid Cableのポーリング間隔を変えて配信レイテンシを実測。1 WebSocketコネクションあたりのメモリ消費を実測。ダッシュボードチャンネルのサンプリング頻度（5秒に1回）の妥当性を負荷データで裏取りする |
| 検証（完了基準） | ブロードキャスト経路の図解が資料なしで書ける。ポーリング間隔とレイテンシの関係が実測値の表で示せる |

### DD-5：手動計装まで下りる（Step 5完了後、インフラ系）

自動計装を入れて満足せず、計装の中身を理解して自前spanを書く。

| 段階 | タスク |
|------|-------|
| 読む | opentelemetry-instrumentation-railsが何をどうパッチしているか |
| 手を動かす | FFmpeg変換ジョブ・R2アップロードに自前spanを仕込む。trace_idをRailsのログと相関させる |
| 検証（完了基準） | FFmpeg変換の内訳（解像度別の変換時間）がGrafanaのトレースで見える |

### DD-9：紙上の設計を実構成に格上げ（Step 9、インフラ系）

Step 9は元々「スケール設計ドキュメントを書く」だったが、一時的に実構成を組んで検証まで行う。

| 段階 | タスク |
|------|-------|
| 手を動かす | 2台目VPSを一時契約し、Web/Solid Queue Workerを分離。Cloudflare LB（またはKamalのmulti-host構成）で振り分け。1台構成と同一の負荷試験を実施し、ボトルネックの移動を観測。Action Cable複数台時のpub/sub共有（Solid CableのDB共有で成立すること）を確認 |
| 記録 | 1台/2台構成の負荷試験結果の比較表。複数台化で踏んだ問題と解決をスケール設計ドキュメントに反映 |
| 検証（完了基準） | 同一負荷試験の1台/2台比較表がある。検証後に2台目を解約し、コストが一時増（+500〜1,000円/月×1ヶ月程度）で収まっている |

### DD-10：障害注入訓練（Step 10完了後、運用系）

障害対応マニュアルを「書いた」で終わらせず、実際に障害を起こして使う。

| 段階 | タスク |
|------|-------|
| 手を動かす | ゲームデイとして3シナリオを実施：①FFmpeg変換の失敗注入、②DB停止、③R2アクセスキー無効化。マニュアルだけを頼りに復旧し、MTTRを記録する |
| 記録 | 各シナリオの復旧手順の実測時間と、マニュアルの不備の修正履歴 |
| 検証（完了基準） | 3シナリオすべてマニュアルのみで復旧できた。マニュアルの不備が最低1件見つかり修正されている（1件も見つからないのは演習の負荷が甘い兆候とみなす） |

## 4. スケジュール改訂

Part 3のスケジュール（約10ヶ月）に深掘り8本（うちStep 3.7は本体Step）を積むと、全体は**12〜13ヶ月級**になる。

```
Month 1      : Step -2 + Step -1
Month 2      : Step 0（復習）+ Step 1 + DD-1
Month 3〜4   : Step 2 + DD-2
Month 4〜5   : Step 3 + DD-3 + Step 3.5
Month 5.5    : Step 3.7（API再設計）
Month 6      : Step 4 + Step 4.5 + DD-4.5
Month 7      : Step 5 + DD-5 + Step 6
Month 8      : Step 7（YouTube API連携）
Month 9      : Step 8（AI機能、任意）
Month 10〜11 : Step 9 + DD-9 + Step 10 + DD-10
Month 12     : Step 11（記事化・ポートフォリオ総仕上げ）
```

短期間で見せられる状態にする場合の優先順位（Part 3記載）は変わらない。その場合、深掘りはDD-1とDD-4.5だけ残して他を後回しにする（差別化の核心である軸2まわりの理解を優先）。

## 6. コストへの影響

- DD-9の2台目VPS：一時的に+500〜1,000円/月（検証後に解約）
- それ以外の深掘りは追加コストなし（負荷試験はローカルまたは既存VPS内で実施）
- 月数千円以内の制約（NFR-5）は維持できる
