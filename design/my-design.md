# Step 1 設計メモ：HLSアーカイブ配信 MVP

## 全体構成

```
[管理者] → Presigned URLで直接アップロード → [Cloudflare R2]
                                                    ↓
                                        Solid Queue（解像度ごとに別ジョブ）
                                        FFmpegでHLS変換
                                                    ↓
                                        [Cloudflare R2 + CDN]
                                                    ↓
                                        [視聴者ブラウザ]（hls.js）
```

---

## Q1. アップロード経路

**結論：Presigned URLによるR2直接アップロードを採用する**

### 判断材料
- ライブアーカイブのMP4は数GB〜10GB程度になりうる（10分の動画で約500MB実測、2時間なら6GB程度と概算）
- Rails経由でアップロードすると、そのリクエストの間Pumaのワーカーが専有される
- 1人の管理者が使うだけなら実害は小さいが、将来的な運用・スケールを見据えて最初から負荷の低い設計にしておく

### 仕組み
```
1. ブラウザ → Rails：「アップロードしたい」とリクエスト（軽量、一瞬で終わる）
2. Rails → ブラウザ：期限付きのPresigned URLを発行
3. ブラウザ → R2：Presigned URLを使い、ファイルを直接アップロード
   （Railsは一切データの中身を経由しない）
```

Railsの役割は「権限確認とURL発行」のみに限定し、実際の重いデータ転送はブラウザとR2の間で完結させる。

## Q2. 変換ジョブの設計

**結論：解像度ごとに独立したジョブとし、べき等性を持たせる**

### ジョブの単位
- 1動画1ジョブではなく、解像度（1080p / 720p / 360p）ごとに別ジョブとする
- 理由：1つの解像度の変換失敗が、他の成功済み解像度の再処理を引き起こさないようにするため

### 失敗時の挙動
- Solid Queueの標準的なリトライ機構に任せる
- 失敗した解像度のジョブだけが再実行され、成功済みの他解像度には影響しない

### べき等性（同じジョブが2回走っても壊れない設計）
```ruby
class TranscodeResolutionJob
  def perform(video_id, resolution)
    # リトライ時も安全なように、まず既存ファイルを削除してから始める
    R2Client.delete_prefix("videos/#{video_id}/#{resolution}/")
    transcode(video_id, resolution)
  end
end
```
- ジョブ実行の最初に対象ディレクトリを削除してから変換を始めることで、何度実行しても同じ最終状態になるようにする
- これにより「途中まで変換されたゴミ」の問題も同時に解決する（次回実行時に削除されるため）

## Q3. R2のディレクトリ構成とキャッシュ戦略

**結論：セグメント（.ts）は長いTTL、マニフェスト（.m3u8）は短いTTLに分ける**

### ディレクトリ構成
```
videos/{video_id}/{resolution}/output.m3u8
videos/{video_id}/{resolution}/output0.ts
videos/{video_id}/{resolution}/output1.ts
...
```

### キャッシュ特性の違い
| ファイル種別 | 中身の性質 | TTL方針 |
|---|---|---|
| .tsファイル（セグメント） | 一度作成したら中身は不変（VODの場合） | 長め（数日〜数週間） |
| .m3u8ファイル（マニフェスト） | VODでは基本的に固定だが、動画差し替え等の運用上「同じURLのまま中身を更新したい」ニーズがある | 短め（数分〜1時間程度） |

### 差し替え時の考慮
- .tsファイルは差し替えたい場合、パスにバージョンを含める運用で回避可能（例：`videos/{video_id}/v2/{resolution}/`）
- .m3u8は同じURLのまま更新したいニーズが強いため、TTLを短めにしておくことで運用上の柔軟性を確保する

## Q4. Videoのモデリング

**結論：statusカラムをenumで管理し、状態を変更できる箇所をコード上で1つに絞る**

### 状態
```ruby
enum status: {
  uploaded: 0,
  transcoding: 1,
  published: 2,
  failed: 3
}
```

### booleanを複数持たない理由
- `uploaded: true, transcoding: true, published: true` のような、本来同時に成立しないはずの組み合わせをDBレベルで防げなくなる
- enumなら1カラムに1値しか入らないため、矛盾した状態がそもそも作れない

### 許されない遷移とその防止策
- 例：`failed` から `published` へ直接遷移してしまうと、公開すべきでない動画（不具合のある素材等）が誤って公開されるリスクがある
- 対策：状態を変更する処理を専用メソッドに集約し、直接 `update` や `update_column` で status を書き換えるコードをどこにも書かない

```ruby
class Video < ApplicationRecord
  def mark_as_published!
    raise "failedからは公開できません" if failed?
    update!(status: "published")
  end
end
```

- 状態遷移が発生する箇所を限定する：
  - `uploaded → transcoding`：アップロード完了時、コントローラーがセット
  - `transcoding → published`：変換ジョブの成功時、ジョブ内の1箇所だけがセット
  - `transcoding → failed`：変換ジョブの失敗時、ジョブ内の1箇所だけがセット
  - `failed → published` という遷移を許すコードは意図的にどこにも書かない

## Q5. 実装の進め方（スライスの切り方）

**結論：Presigned URL直アップロードの技術検証を最優先で行う**

### 理由
- Q2（Solid Queueでのジョブ設計）、Q3（キャッシュ戦略）、Q4（状態遷移）は、いずれもRailsの標準機能やSection 5で動作確認済みのFFmpeg知識の応用であり、「動くかどうか分からない」要素が少ない
- Presigned URLによる直接アップロードのみ、今回のプロジェクトで一度も手を動かして検証していない領域（CORS設定、署名生成、アップロード進捗の扱い等で詰まる可能性がある）

### 進め方
1. 最初にPresigned URL経由でファイルを1個アップロードするだけの最小コードを書き、動作を確認する
2. 動くと確認できてから、Q2〜Q4で設計した内容を積み上げていく
3. 全設計を完成させてから実装に入るのではなく、一番不確実な部分を先に小さく検証する
