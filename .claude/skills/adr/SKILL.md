# ADR（設計判断の記録）

## Overview

「面接でなぜこの設計かを一貫して説明できること」がこのプロジェクトの成功基準（Part 1）。判断した瞬間に記録しないと理由は消える。`docs/adr/` に連番で保存する。

## 手順

1. `docs/adr/` の既存ファイルを確認し、次の連番を採る（初回は `0001`）
2. `docs/adr/NNNN-<スラッグ>.md` を以下のテンプレートで作成する
3. 決定がroadmapの記載と食い違う場合は、その旨をContextに明記する（roadmap自体の更新はユーザーに確認）

## テンプレート

```markdown
# NNNN. <決定内容を一文で>

- Status: accepted
- Date: YYYY-MM-DD
- Step: <該当するroadmap Step>

## 結論

<何をどうすると決めたか。1〜3文>

## Context

<なぜこの判断が必要になったか。制約（コスト・学習目的・個人開発）も含める>

## 判断軸

<適用した軸。データ整合性なら「表示用かトリガーか」（CLAUDE.md 設計原則2）。
軸2関連なら「視聴体験に影響しないか」（設計原則1）>

## 検討した選択肢

| 選択肢 | 利点 | 欠点 | 不採用理由 |
|--------|------|------|-----------|

## 影響

<この決定で生じる制約・将来の変更コスト>

## 面接での一言説明

<30秒で語るならこう言う、を1段落で>
```

## Common Mistakes

- 採用案だけ書いて不採用案を書かない → 「なぜ他ではないか」が語れないADRは半分の価値しかない
- 実装後にまとめて書こうとする → 判断の直後に書く。鮮度が理由の解像度を決める
