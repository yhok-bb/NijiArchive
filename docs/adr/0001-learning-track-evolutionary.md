# 0001. 学習トラックは進化型（MVP lean → Step完了後に深掘りフェーズ）で組み込む

- Status: accepted
- Date: 2026-07-07
- Step: Step -1（要件定義・設計）/ roadmap Part 4作成時

## 結論

Railsエコシステム・ミドルウェア・インフラ・API設計の学習要素は、各Stepの構成自体には組み込まず、Step完了後の「深掘りフェーズ」（OSS読解・置き換え実装・計測実験）として付加する。配分は「1 Step = 1深掘りテーマ」とし、本体Stepの完了基準を満たすまで深掘りに着手しない。詳細は roadmap Part 4。

## Context

このプロジェクトの主目的は学習であり、開発者は実務レベルの難易度・OSSに深く入り込む経験を求めている。一方でスコープ肥大化が「このプロジェクト最大のリスク」（CLAUDE.md）であり、学習密度を上げるほど「完成しない」リスクが高まる。学習密度と完成の両立方法を決める必要があった。

## 判断軸

「完璧より完成」（動くものを先に確保する）と「実務で最も価値のある経験は何か」の2軸。動いているものを安全に置き換える経験は、最初から正しく作る経験より実務転用性が高い、という判断を優先した。

## 検討した選択肢

| 選択肢 | 利点 | 欠点 | 不採用理由 |
|--------|------|------|-----------|
| 進化型（採用） | 動くものが先に出る。「動いているものを安全に置き換える」実務経験が各Stepで積める。深掘りが遅延しても本体は無傷 | 一度作ったものを作り直す工数が発生する | — |
| 最初から本格構成 | 学習密度が最大。作り直しがない | 動くものが出るまでが遠く、モチベーション低下・スコープ肥大化と正面衝突 | プロジェクト最大のリスクを直接増やすため |
| タイムボックス制（深掘り各1週間上限） | スケジュールが守りやすい | OSS読解を時間で区切ると読みかけで終わり、学びが定着しない | 学習という主目的を損なうため |

## 影響

- 全体スケジュールが約10ヶ月→12〜13ヶ月級に伸びる（Part 4に改訂スケジュール）
- 深掘りフェーズにも数値・観察可能な完了基準を課す運用が必要になる
- Step 3.7（API再設計）が本体Stepとして新設され、Step 1〜3のAPIは意図的に「後で置き換える」前提で作ることになる
- DD-9で一時的にVPS2台目のコスト増（+500〜1,000円/月×1ヶ月程度、NFR-5の範囲内）

## 面接での一言説明

学習目的の個人開発で一番怖いのは「完成しないこと」だったので、各機能はまずleanに作って動かし、完了後に深掘りフェーズでOSSの内部を読んでから自分の実装に置き換える、という進化型で組みました。実務で価値があるのは「動いているものを安全に置き換える」経験だと考えたからで、たとえばAPIはStep 1〜3で素朴に作った後、Step 3.7でOpenAPI駆動に再設計する構成にしています。
